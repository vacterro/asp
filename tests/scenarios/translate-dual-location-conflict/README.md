Test: a project carries translation output at BOTH root-level `.saitranslate/` (legacy, pre-v7.35.0) AND `.saipen/saitranslate/` (current) simultaneously -- a partial/failed migration, or two agents disagreeing about which convention to use (the exact class of situation a real cross-project test surfaced: one agent mid-migrating while another instance was still active against the old path). Per RFC § 2.1's dual-location clause (v7.37.0, mirroring § 1.9's identical rule for extensions), the agent MUST NOT silently merge the two, guess which is newer, or pick one at random. Correct behavior: treat `.saipen/saitranslate/` as authoritative, the root-level copy as stale, ticket its removal (a normal `TODO`, not urgent -- TRANSLATE proceeds using the authoritative copy), and continue. The failure mode this guards: an agent reading half of one copy and half of the other, silently producing a translation bundle that mixes two generations of source drift with no signal anything is off. Behavioral test -- the assertion is which copy wins and that the conflict gets flagged, not a specific file shape.
