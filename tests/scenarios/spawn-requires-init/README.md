Test: the user (or another agent) tries to start work that assumes `.saipen/` already exists -- either `saipen sub spawn <name>` (RFC § 1.9, v7.36.0) or a separate, dedicated agent instance starting a parallel TRANSLATE build (RFC § 2.1, v7.37.0) -- on a project that has no `.saipen/` at all yet (`saipen set` never ran). Correct behavior in both cases: refuse, and tell the user to run `saipen set` first -- MUST NOT silently trigger `INIT` as a side effect of an unrelated command, and MUST NOT improvise a bootstrap of its own. The failure mode this guards: a subSaipen or a parallel TRANSLATE instance inventing its own ad-hoc `.saipen/`-shaped folder to have somewhere to write, producing a state file that `saipen continue` was never designed to find or trust, diverging from the one true bootstrap path INIT already owns. Behavioral test -- the assertion is that the agent refuses and names the right command, not a specific file shape.
